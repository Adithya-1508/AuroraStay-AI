# Definition of Ready (DoR)

Before any developer (or agent) begins implementation on a feature, it must meet the Definition of Ready to prevent scope creep, architectural misalignment, or code duplication.

## Readiness Checklist

- [ ] **Business Objective**: A clear, measurable business goal is defined (what problem does this solve?).
- [ ] **RFC**: A Request for Comments has been written, reviewed, and approved.
- [ ] **Acceptance Criteria**: Concrete, testable conditions of satisfaction are documented.
- [ ] **API Design** (if applicable): OpenAPI endpoints, schemas, and methods are defined.
- [ ] **Data Model** (if applicable): Database schema modifications, entities, and relationships are designed.
- [ ] **Test Strategy**: Unit, integration, and evaluation strategies are planned.
- [ ] **Dependencies Identified**: Internal and external libraries, resources, and systems are mapped.
- [ ] **Approved Specification**: A file exists under `.specs/` detailing the design.
